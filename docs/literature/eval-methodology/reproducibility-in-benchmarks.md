# Reproducibility and Rigor in AI Benchmarks

## Why Reproducibility Is Hard for LLM Benchmarks

Unlike traditional software benchmarks, LLM benchmarks face unique reproducibility threats:

1. **Benchmark contamination**: training data for frontier models almost certainly includes benchmark questions, inflating performance on "held-out" test sets
2. **Train-test leakage**: when benchmark data is publicly available, it migrates into pre-training corpora over time
3. **Non-determinism**: even at temperature=0, API-based models can produce different outputs across versions or infrastructure updates
4. **Evaluation drift**: as the benchmark community accumulates insight, evaluation protocols evolve, making longitudinal comparison difficult

---

## Benchmark Contamination

### The Problem
Models trained on internet-scale corpora may have seen benchmark questions during pre-training. This is especially severe for:
- Static text-based benchmarks (MMLU, BIG-Bench) that were published years before current model training cutoffs
- Benchmarks where the answer format is highly distinctive (multiple choice, structured outputs)

### Evidence
- Models show performance cliffs on rephrased or shuffled variants of benchmark questions (suggesting memorization rather than generalization)
- Codex-class models score 10–30% higher on HumanEval than on structurally equivalent but unfamiliar coding problems
- Golchin & Surdeanu (2023) demonstrate that GPT-4 reproduces benchmark examples verbatim from training data

### Mitigations
- **Dynamic evaluation**: generate new tasks from templates/parameters; scores cannot be memorized
- **Temporal holdout**: use only tasks created after a model's training cutoff
- **Closed/private test sets**: separate public development set from private evaluation set (SWE-bench Verified uses this approach)
- **Semantic perturbation**: systematically paraphrase tasks; check if performance degrades significantly

---

## Static vs. Dynamic Benchmarks

| Type | Example | Contamination Risk | Adaptability |
|------|---------|-------------------|--------------|
| Static text | MMLU, BIG-Bench | High — questions published online | Low |
| Static code | HumanEval, MBPP | High — problems in training data | Low |
| Dynamic template | LiveBench, BuyerBench scenario variants | Low — parameters instantiated at eval time | High |
| Live environment | WebArena, SWE-bench live | Low — real environment state changes | Medium |

### LiveBench (2024)
Generates new benchmark questions monthly from recent events; avoids contamination by design. Trade-off: harder to track longitudinal progress.

---

## Snapshot vs. Live Evaluation

**Snapshot evaluation** uses a fixed dataset frozen at a point in time. Advantages: easy to share, cite, and reproduce exactly. Disadvantages: becomes contaminated over time; doesn't capture model capability improvements in context.

**Live evaluation** continuously generates or updates tasks. Advantages: contamination-resistant; tracks real-world performance. Disadvantages: harder to compare papers citing different evaluation snapshots; requires maintained infrastructure.

### BuyerBench's Approach
BuyerBench uses a **controlled variant design**: each scenario has a fixed structure (task objective, constraints, evaluation criteria) but parameterized economics (prices, quantities, supplier identities). This allows:
- New scenario instances to be generated by re-parameterizing an existing scenario structure — contamination-resistant
- Exact replay of a specific parameter instance for reproducibility in reported results
- Controlled perturbation variants (same economics, different framing) that cannot be "pre-answered" by training data because the framing change is applied at evaluation time

This is a middle ground between static snapshots (easily contaminated) and fully live evaluation (hard to reproduce): **reproducible-but-fresheable scenarios**.

## Variant-Based Reproducibility in BuyerBench

The core approach to Pillar 2 behavioral bias testing is directly relevant to reproducibility:

```
Scenario A: "Supplier X offers the lowest total cost. Select."
Scenario A': "Supplier X costs $50 more than a competitor but includes free delivery — is it worth it?"
```

Scenarios A and A' present identical economics but different framings. An agent that gives inconsistent answers is susceptible to framing bias. This technique:
- Cannot be defeated by memorization (the framing is injected at test time)
- Produces a reproducible test pair (the underlying economics are versioned)
- Generates its own internal consistency check (no external oracle needed)

## Evaluation Rigor Best Practices

1. **Version-stamp everything**: record model version, temperature, seed, evaluation harness version, and timestamp for every evaluation run
2. **Separate dev and test sets**: agents/researchers should tune against dev scenarios; final scores use sealed test scenarios
3. **Report variance, not just means**: run multiple trials; report confidence intervals or standard deviation across runs
4. **Document human baseline**: if human procurement specialists would score 90%, an agent at 75% is materially below baseline; an agent at 95% is materially above
5. **Publish evaluation code**: evaluation logic should be open-source and versioned alongside benchmark data
